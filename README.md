# QuillQR

QuillQR is a small educational web app for creating simple QR-linked
documents.

The normal public workflow is text/Markdown. A separate trusted PDF lane
can be enabled for PDF-only uploads, with sanitized public PDF output.

This project is intentionally small, readable, and practical. It is not a
framework, platform, or polished commercial product. It is a learning
codebase and a working prototype.

## Current State

QuillQR currently supports:

- creating public text or Markdown documents
- generating QR codes for public document links
- secret edit links for updating existing documents
- local SQLite storage
- Flask + Gunicorn deployment behind Nginx
- example systemd and Nginx deployment files
- optional trusted PDF upload lane
- PDF-only upload validation
- Ghostscript-based PDF cleanup/sanitization
- selectable-text checks for PDFs
- inline browser PDF viewing

The PDF upload lane is separate from the normal document lane:

- `/d/<code>` is for normal public text/Markdown documents
- `/e/<secret>` is for normal document editing
- `/p/<secret>` is for trusted PDF upload/create mode
- `/p/d/<code>` is for public PDF documents
- `/p/e/<secret>` is for PDF replacement/re-upload
- `/p/file/<code>` serves the sanitized PDF inline

PDF uploads are disabled by default in production examples.

## PDF Safety Model

The current PDF safety model is:

1. accept PDF files only
2. reject non-PDF uploads
3. store uploads outside the public static directory
4. rewrite/sanitize PDFs with Ghostscript
5. reject encrypted PDFs
6. reject unsupported active PDF features where detected
7. require selectable text on a configurable percentage of pages
8. serve the sanitized public copy, not the original upload

Antivirus scanning was tested, but is currently disabled in the example
production configuration because ClamAV can be too heavy for a 1 GB VPS.
PDF cleanup/sanitization is currently the primary protection layer.

## Development Method

QuillQR was built as an AI-assisted programming project.

Most of the implementation code was generated by ChatGPT under active
human guidance, testing, correction, deployment, and review by Hammer
Tomahawk.

This was not a simple “make me an app” prompt. The project was developed
through an iterative technical conversation: architecture decisions,
security tradeoffs, deployment behavior, PDF handling, systemd hardening,
Nginx configuration, debugging, and production constraints were actively
directed and tested by the human developer.

A fair description is:

> AI-assisted implementation with human technical direction.

Or, less formally:

> vibe coding with supervision, corrections, and actual engineering
> judgment.

The result is intentionally preserved as an educational codebase.

## Deployment Shape

Recommended deployment layout:

```text
/home/<user>/repos/quillqr
  Git checkout / authoring copy

/srv/quillqr
  runtime deployment copy
  no .git directory
  no repository history
```

The `/srv/quillqr` directory should be a clean deployment copy, not the
Git repository itself.

Example deployment files live under:

```text
deploy/
```

These are templates and should be edited for the target server.

## Runtime Notes

QuillQR does not currently require user accounts.

The app does not intentionally set cookies in the current flow. Server
access logs may still contain normal web server request information such
as IP address, path, timestamp, and user agent.

Some frontend dependencies may be loaded from public CDNs depending on
the current templates. For stricter privacy and production control, those
files should eventually be vendored into `/static/vendor/`.
